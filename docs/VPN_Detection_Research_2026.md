## Executive Summary

**Ситуация переломная.** 2 апреля 2026 Минцифры провело закрытые совещания с 20+ крупнейшими российскими платформами и поставило дедлайн: **к 15 апреля 2026** внедрить детекцию и блокировку VPN-пользователей. 6 апреля разослана техническая методичка. 3–4 апреля агрессивные блокировки ТСПУ случайно уронили Сбербанк, ВТБ и Т-Банк — ущерб свыше $1 млрд.

Угроза для VPN-пользователей двухслойная:
1. **Сетевой уровень (ТСПУ/DPI):** блокирует OpenVPN, WireGuard, IKEv2; Reality на 443 нестабилен; AmneziaWG 2.0 — наиболее устойчив
2. **Уровень приложений:** методичка описывает `ConnectivityManager.TRANSPORT_VPN` как главный метод — нескрываемый без root; плюс сканирование xray gRPC API на localhost

**Ключевые факты для разработчика:**
- `TRANSPORT_VPN` нельзя скрыть без root — это архитектурное ограничение Android VpnService
- xray gRPC API (порт 10085) по умолчанию открыт без аутентификации — критическая уязвимость
- `/proc/net/` закрыт с Android 10+ через SELinux — этот вектор мёртв
- VLESS + Reality + нестандартный высокий порт — лучшее против ТСПУ на сегодня
- Cloudflare CDN-фронтинг мёртв в России с середины 2025

## Блок 2. Android API методы детекции VPN

### 2.1 ConnectivityManager.getNetworkCapabilities() → TRANSPORT_VPN ⭐ ГЛАВНЫЙ

**Требования:** только `ACCESS_NETWORK_STATE` (normal protection level, выдаётся без диалога пользователю). API 21+ (Android 5.0).

```java
ConnectivityManager cm = context.getSystemService(ConnectivityManager.class);
NetworkCapabilities caps = cm.getNetworkCapabilities(cm.getActiveNetwork());

// Метод 1: прямая проверка
boolean isVPN = caps != null && caps.hasTransport(NetworkCapabilities.TRANSPORT_VPN);

// Метод 2: через отсутствие NOT_VPN capability
boolean notVPN = caps != null && caps.hasCapability(NetworkCapabilities.NET_CAPABILITY_NOT_VPN);
// Если notVPN == false → VPN активен

// Метод 3: перебор всех сетей
for (Network net : cm.getAllNetworks()) {
    NetworkCapabilities nc = cm.getNetworkCapabilities(net);
    if (nc.hasTransport(TRANSPORT_VPN)) { /* VPN обнаружен */ }
}
```

**Что раскрывает:**
- Факт активного VPN на уровне всей системы
- Работает даже при split-tunnel: флаг установлен если VPN-интерфейс активен, даже если трафик конкретного приложения идёт в bypass

**Обходится ли без root:** НЕТ. Только через Xposed/LSPosed (модуль `NoVPNDetect`). Это фундаментальная архитектурная особенность Android — флаг выставляется ядром через `ConnectivityService`.

**Вывод:** нескрываемый без root метод. Любое Android-приложение из Play Store обнаружит VPN через этот вызов.

### 2.2 VpnTransportInfo / VcnTransportInfo

**API level:** `VpnTransportInfo` с API 31 (Android 12). `getTransportInfo()` — с API 29.

```java
NetworkCapabilities caps = cm.getNetworkCapabilities(network);
TransportInfo transportInfo = caps.getTransportInfo(); // API 29+
if (transportInfo instanceof VpnTransportInfo) {
    VpnTransportInfo vpnInfo = (VpnTransportInfo) transportInfo;
    int vpnType = vpnInfo.getType();
    // VpnManager.TYPE_VPN_SERVICE   — пользовательский VPN (v2rayNG, Hiddify, etc.)
    // VpnManager.TYPE_VPN_PLATFORM  — MDM/enterprise VPN
    // VpnManager.TYPE_VPN_LEGACY    — PPTP/L2TP через Settings
}
```

**Что раскрывает:** тип VPN — пользовательский (`TYPE_VPN_SERVICE` = все v2rayNG/Hiddify/NekoBox) vs. корпоративный. Позволяет отличить пользовательский VPN от корпоративного и не блокировать корпоративных пользователей.

### 2.3 NetworkInterface.getNetworkInterfaces() — имена интерфейсов

**Требования:** разрешений не нужно. API 1+.

```java
Enumeration<NetworkInterface> interfaces = NetworkInterface.getNetworkInterfaces();
while (interfaces.hasMoreElements()) {
    NetworkInterface iface = interfaces.nextElement();
    String name = iface.getName().toLowerCase();
    if (name.startsWith("tun")   ||  // xray TUN, sing-box TUN, Clash TUN, OpenVPN
        name.startsWith("tap")   ||  // OpenVPN TAP
        name.startsWith("ppp")   ||  // PPTP, L2TP
        name.startsWith("wg")    ||  // WireGuard (wg0, wg1...)
        name.startsWith("ipsec")) {
        // VPN обнаружен
    }
    // Дополнительные флаги:
    boolean isP2P = iface.isPointToPoint(); // характерно для tun/ppp
    int mtu = iface.getMTU();               // для MTU-анализа
}
```

**Обходится ли:** только через Xposed (подмена списка интерфейсов) или root (переименование через `ip link`). Переименование через `VpnService.Builder.setSession()` не влияет на имя в ядре.

### 2.4 getifaddrs() через JNI/NDK

**Доступность:** NDK с API 24 (Android 7.0, Bionic libc).

Предоставляет те же данные что и Java `NetworkInterface`, но через C-код:
- Прямой доступ к флагам ядра: `IFF_POINTOPOINT`, `IFF_NOARP`, `IFF_MULTICAST`
- Статистика трафика через `ifa_data`
- Полезен в native-коде антифрод SDK

**Практически:** Java API использует тот же `getifaddrs()` под капотом. Принципиальной разницы для детекции нет, но SELinux не защищает от NDK вызовов — работает на уровне ядра.

### 2.5 MTU-анализ

**Характерные MTU значений VPN-протоколов:**

| Протокол | MTU внутри туннеля | Примечание |
|---|---|---|
| Ethernet (стандарт) | **1500** | |
| WireGuard (IPv4 uplink) | **1420** | 1500 − 80 байт overhead |
| WireGuard (IPv6 uplink) | **1400** | Больший overhead IPv6 |
| AmneziaWG | **1280** (рекомендуется) | Намеренно нестандартный |
| xray TUN (VLESS) | 1500 или кастомный | Зависит от конфига |
| sing-box TUN | 1500 (default) | Настраивается в `tun.mtu` |
| OpenVPN UDP | ~1400–1450 | Зависит от cipher |

```java
NetworkInterface iface = NetworkInterface.getByName("tun0");
int mtu = iface.getMTU(); // 1420 → WireGuard
```

**На сетевом уровне (серверная сторона):** MTU виден в TCP SYN пакетах как MSS = MTU − 40. Инструмент: `p0f-mtu` (ValdikSS/p0f-mtu на GitHub).

### 2.6 /proc/net/route — УСТАРЕЛ

**Статус Android 10+:** полностью заблокирован через SELinux-политику `proc_net`.

- До Android 9: любое приложение могло читать `/proc/net/tcp` — видело все соединения включая `127.0.0.1:10085` xray
- Android 10+ (targetSdk ≥ 29): `Permission denied` для домена `untrusted_app`
- Исключения (доступ сохраняется): системные приложения, platform-key подпись, VPN-домен

**Вывод:** метод мёртв для приложений из Play Store targeting API 29+. SELinux работает на уровне ядра и блокирует вне зависимости от способа вызова (включая NDK).

### 2.7 dumpsys vpn_management — НЕДОСТУПЕН

```java
// Любая попытка завершится с SecurityException:
Process p = Runtime.getRuntime().exec("dumpsys vpn_management");
// SecurityException: requires android.permission.DUMP (signatureOrSystem)
```

**Причина:** `android.permission.DUMP` требует platform-ключ подписи. Из-за этого методичка Минцифры ОШИБОЧНО указывает этот метод как доступный для обычных приложений.

**Что возвращает через ADB:** состояние VPN по UID, имя VPN-приложения, состояние соединения.

### 2.8 Системные переменные http.proxyHost / socksProxyHost

```java
String httpProxy  = System.getProperty("http.proxyHost");
String socksHost  = System.getProperty("socksProxyHost");
boolean proxySet  = httpProxy != null || socksHost != null;
// 127.0.0.1:10809 → xray HTTP proxy активен
// 127.0.0.1:7890  → Clash HTTP proxy активен
```

**Ограничения:** устанавливается только в proxy-mode (без TUN). В TUN-режиме эти переменные не устанавливаются. Многие нативные приложения игнорируют системные прокси.

### 2.9 Активные методы — сканирование localhost ⭐ ВЫСОКАЯ УГРОЗА

**Без каких-либо разрешений** любое приложение может сканировать `127.0.0.1`:

```java
int[] ports = {10808, 10809, 7890, 7891, 2334, 2333, 1080, 10085, 9090};
for (int port : ports) {
    try {
        Socket s = new Socket();
        s.connect(new InetSocketAddress("127.0.0.1", port), 200);
        s.close();
        // Порт открыт → VPN-клиент активен
    } catch (Exception ignored) {}
}
```

| v2rayNG | 10808 | SOCKS5 |
| v2rayNG | 10809 | HTTP |
| Hiddify | 2334 | SOCKS5 |
| Hiddify | 2333 | HTTP |
| NekoBox | 2334 | SOCKS5 |
| Clash for Android | 7891 | SOCKS5 |
| Clash for Android | 7890 | HTTP |

### 2.10 xray gRPC API — КРИТИЧЕСКАЯ УЯЗВИМОСТЬ ⭐

**Уязвимость:** xray gRPC API по умолчанию **не имеет аутентификации**. Порт 10085 открыт любым приложением на устройстве.

```bash
# Атакующее приложение может выполнить:
grpcurl -plaintext 127.0.0.1:10085 list
# → xray.app.proxyman.command.HandlerService
# → xray.app.stats.command.StatsService

grpcurl -plaintext -d '{}' 127.0.0.1:10085 \
  xray.app.proxyman.command.HandlerService/GetInboundUsers
```

**Что раскрывает:**
- Факт работы xray-core (любой xray-based клиент)
- Список протоколов (VLESS, VMess, Trojan, Shadowsocks)
- **IP/домен VPN-сервера** в конфигурации outbound
- Статистику трафика

**Другие уязвимые порты:** 19085 (Marzban-панели), 23456 (Hiddify-специфичные конфиги).

### 2.11 Clash/Mihomo REST API — аналогичная уязвимость

### 2.12 SOCKS5 handshake идентификация

```java
Socket s = new Socket("127.0.0.1", port);
s.getOutputStream().write(new byte[]{0x05, 0x01, 0x00}); // SOCKS5 greeting
byte[] resp = new byte[2];
s.getInputStream().read(resp);
if (resp[0] == 0x05) { /* SOCKS5 подтверждён */ }
```

## Блок 3. Конкретные приложения

### 3.1 VK MAX — задокументированная активная детекция

**Наиболее задокументированный случай.** Март 2026, реверс-инжиниринг APK `MAX_(RS)_v.26.4.3`:

- Приложение регулярно отправляет HTTP-запросы к списку сайтов (~половина российские, половина заблокированные в РФ)
- **Анализ доступности этих ресурсов = косвенная детекция VPN**
- Данные смешиваются с основным трафиком — невозможно заблокировать аналитику без блокировки самого приложения
- Собирает PLMN (`'operator': '25001:MTS'`) из текущей SIM-карты
- Функция `callDontUseVpnForRtp` настраивает split tunneling через нативный C-API

**Источник на ntc.party:** тред 22584 (5 марта 2026, 266 постов): «TL;DR — содержит шпионские модули для слежки за пользователями VPN и удалённого управления ими».

**Протокол передачи данных:** не HTTP/HTTPS, не gRPC — проприетарный бинарный TCP-протокол с полем PLMN, команды длиной 10 байт.

### 3.2 Банки — инцидент 3–4 апреля 2026 ⭐

**Критическое событие:** агрессивные ТСПУ-блокировки VPN **случайно задели банковскую инфраструктуру**:
- Упали: Сбербанк, ВТБ, Т-Банк
- Перестали работать терминалы, банкоматы, мобильные приложения
- Московское метро открыло турникеты
- **Ущерб: свыше $1 млрд**
- Павел Дуров подтвердил: причина — VPN-блокировки

**Причина:** TLS-трафик банков внешне неотличим от VPN-трафика → ТСПУ ошибочно попал банковские IP в фильтр.

**Т-Банк:** официальная справка прямо рекомендует отключить VPN при входе в личный кабинет.

**Сбербанк:** активной программной VPN-детекции в приложении не зафиксировано — блокировка идёт через ТСПУ. Присутствовал на совещании Минцифры — обязан внедрить к 15 апреля.

### 3.3 Маркетплейсы

**7 апреля 2026:** Wildberries и Ozon начали нестабильно работать при включённом VPN — не загружаются карточки, изображения, оформление заказа затруднено. ВкусВилл — в той же волне.

**Авито:** присутствовал на совещании Минцифры 28 марта, требования получены.

**Механизм:** IP пользователя сопоставляется с реестром VPN-адресов РКН.

### 3.4 Яндекс / Кинопоиск

Присутствовал на совещании, обязан внедрить детекцию к 15 апреля. Кинопоиск входит в экосистему Яндекса — подпадает через материнскую компанию. Конкретных публичных анализов трафика нет.

### 3.5 Операторы связи

Операторы — **инфраструктурный слой**, детекция через ТСПУ, а не через приложения:
- МТС, Билайн, МегаФон, Ростелеком: используют DPI-оборудование ТСПУ
- Проверки март 2026: все четыре блокируют OpenVPN и стандартный WireGuard
- AmneziaWG держится стабильнее всего у МТС и Билайн

## Блок 4. ТСПУ и сетевой уровень детекции

### 4.1 Текущее состояние блокировок (апрель 2026)

| Протокол | Статус | Примечание |
|---|---|---|
| OpenVPN UDP/TCP | ❌ Заблокирован | Стабильно у всех операторов |
| WireGuard стандартный | ❌ Заблокирован | По UDP-сигнатурам на погранузлах |
| IKEv2, L2TP, PPTP | ❌ Заблокированы | |
| Shadowsocks plain | ❌ Заблокирован | В течение часов после запуска |
| VMess без маскировки | ⚠️ Нестабильно | Блокируется часто |
| VLESS + Reality (порт 443) | ⚠️ Нестабильно | Мгновенные блокировки/замедления |
| VLESS + Reality (высокий порт) | ✅ ~80% пакетов | ТСПУ не инспектирует все порты |
| SOCKS5, SSH | ⚠️ Блокировки с дек. 2025 | |
| AmneziaWG 2.0 | ✅ Наиболее устойчив | Старые версии AmneziaWG блокируются |
| XHTTP | ✅ Перспективный | Следующий вектор обхода |

**Статистика:** к марту 2026 — **469 заблокированных VPN-сервисов** (рост +70% за 3 месяца, было 258 в октябре 2025).

### 4.2 Архитектура ТСПУ (из ntc.party, тред 22307)

Четыре компонента системы:
1. **Коллектор** — установлен в разрыв между сетью провайдера и интернетом; режимы: Power-off Bypass, Active
2. **Маршрутизатор (EcoHighway)** — 3.2 Гбит/с, 32 Гбит; фильтрация по 5-tuple (SrcIP+DstIP+Protocol+SrcPort+DstPort); не может делить TCP/UDP-сессии по разным фильтрам
3. **Фильтр (EcoFilter)** — Intel Xeon; DPI L4-L7, сигнатурный анализ, отслеживание TCP-сессий; работает на L2
4. **Сервер правил** — VPN-узел связи с центром, сервер SPFS (предварительное формирование списков), SPHD

**Режимы работы:**
- **Тип A (In-line):** в разрыв от сети
- **Тип B (Ответвление):** включен между крупными пирами в точках обмена; трафик попадает по спискам IP/портов через BGP или Port Redirect

**Ключевой вывод от пользователя ntc.party `0ka`:** «Цель известна заранее по заметкам о структуре трафика. Если их 15 минут — это примерно».

### 4.3 Четыре уровня детекции DPI

**Уровень 1 — Сигнатурный анализ:**
Сравнение первых 16–32 байт с паттернами. OpenVPN, Shadowsocks, WireGuard заваливаются здесь. VLESS с TLS проходит — первые байты идентичны обычному HTTPS.

**Уровень 2 — TLS Fingerprinting (JA3/JA4):**
Анализ параметров ClientHello: cipher suites, extensions, elliptic curves. JA4 (2023) устойчивее к обфускации чем JA3. Когда User-Agent говорит «я Chrome», но JA3-хеш соответствует OpenVPN или Python — детекция мгновенная.

**Уровень 3 — Active Probing:**
ТСПУ отправляет тестовые соединения с разными TLS-версиями. Легитимные сайты отвечают нормально. Подозрительный ответ — триггер блокировки. Из ntc.party (тред 22934): ТСПУ модифицирует ClientHello — меняет TOS (0x00 → 0x28), длины TLS/IP, Client Random, Cipher Suites. Reality обнаруживает: `REALITY: processed invalid connection: failed to read client hello`.

**Уровень 4 — Поведенческий анализ (ML):**
Reality маскирует установку соединения, **но не трафик после handshake**. ML, обученная на реальном Apple-трафике (за который Reality себя выдаёт), видит аномалию. Серверы с высоким трафиком блокируются быстрее.

### 4.4 Детали блокировки по протоколам (ntc.party, тред 5124)

**Метод детекции для каждого протокола:**

| Протокол | Метод детекции | Действие |
|---|---|---|
| L2TP | Детект L2TP Control Message | Пропуск к серверу |
| IPsec | Детект после нескольких пакетов установки сессии | Блокировка UDP |
| PPTP | Детект ответа Start-Control-Connection-Reply | TCP RST после первого пакета |
| OpenVPN UDP | Детект DATA-пакетов после хендшейка | Блокировка UDP после нескольких DATA |
| OpenVPN TCP | Детект DATA-пакетов после хендшейка | TCP RST после нескольких DATA |
| WireGuard | Детект Transport Data на сервере | Блокировка UDP после 5 минут Data-пакетов |

**Ключевое наблюдение ValdikSS (ntc.party):** для всех VPN-протоколов кроме L2TP — сначала выполняется обнаружение протокола, затем «проверка» (probe — попытка ответить на несколько пакетов в TCP/UDP-сессии), затем разрыв сессии. Сессия по 4-tuple (srcip-srcport-dstip-dstport) добавляется в blocklist.

**Критически важно:** ТСПУ добавляет сессию в allowlist **после ~6 первых пакетов**. DPI не может смотреть в середину сессии — ищет только в начале. Метод `bolvan` (nfqws с fake-пакетами): 6 fake-пакетов с тем же IP/портом отправляются сначала → DPI «видит» 6 пакетов «соединения», добавляет в allowlist → настоящие пакеты проходят.

### 4.5 WireGuard — специфика детекции

WireGuard обнаруживается по `message_type = 1` (Handshake Initiation) во входящих пакетах. **AmneziaWG 2.0** заменяет magic headers на случайные (H1-H4) и добавляет junk-пакеты — DPI не может написать универсальное правило.

Из ntc.party (тред про МТС, март 2026): «AWG не виден в wireshark как WireGuard, хотя и с кривыми пакетами» + «Определённая часть трафика каждого пользователя даже с правильным AWG... Если 87 рублей в день слишком дорого, не надо рассчитывать».

### 4.6 Reality/XTLS — почему сложнее и что уже обнаруживается

**Почему сложнее:**
- Не создаёт собственный TLS-стек — использует реальный сертификат и параметры целевого сервера (например, icloud.com)
- Установка соединения неотличима от легитимного TLS-хендшейка

**Текущие уязвимости (2025–2026):**
1. **TLS fingerprint:** с апреля 2025 смена fingerprint перестала помогать на большинстве конфигов. Из ntc.party (тред 16061): «Все фингерпринты chrome/android/etc уже не помогают»
2. **Поведение после handshake:** Reality маскирует только установку. После — паттерн трафика аномален
3. **Active probing:** ТСПУ модифицирует ClientHello (тред 22934) — Reality обрывает соединение при обнаружении нарушений: `failed to read client hello`
4. **Порт 443 — мгновенная блокировка/замедление**; тот же конфиг на высоком порту — значительно стабильнее

**uTLS (ntc.party, тред 21834):** разработчик sing-box прямо заявляет: `uTLS is not recommended for censorship circumvention due to fundamental architectural limitations`.

### 4.7 Cloudflare CDN-фронтинг — мёртв в России с 2025

| Дата | Событие |
|---|---|
| Ноябрь 2024 | РКН рекомендовал российским ресурсам отказаться от Cloudflare + запретить TLS ECH |
| Февраль 2025 | РКН принудительно внёс Cloudflare в реестр ОРИ |
| Апрель 2025 | РКН запретил использование TLS ECH в Cloudflare |
| Июнь 2025 | Основные IP-диапазоны Cloudflare добавлены в реестр заблокированных → трафик из России упал на ~50% |

**Ntc.party (тред 12681, декабрь 2024):** «gRPC через CloudFlare перестал работать сразу на всех устройствах с 4-мя провайдерами».

**Вывод:** CDN-фронтинг через Cloudflare неработоспособен в России с середины 2025 года.

### 4.8 Корреляционная детекция (ntc.party, тред 7318, февраль 2024)

ValdikSS описывает метод с trigg-списками: если пользователь посетил известный VPN-сервис, а затем в течение 5 минут установил OpenVPN или WireGuard с неизвестного IP — этот IP высоковероятно является VPN-сервером и блокируется.

### 4.9 Энтропийный анализ

Из академических исследований (USENIX Security 2024): система **dMAP** — пассивный мониторинг, fingerprint-рассогласования между слоями. Точность 95% на Shadowsocks и VMess.

ТСПУ применяет энтропийный анализ: блокирует «неидентифицированные» зашифрованные соединения с высокой энтропией, не похожие на браузерный трафик.

## Блок 5. Защита — контрмеры против каждого метода

### 5.1 TRANSPORT_VPN — что нельзя скрыть без root

| Что нельзя скрыть | Почему |
|---|---|
| `TRANSPORT_VPN` flag | Выставляется ядром через `ConnectivityService` |
| `NET_CAPABILITY_NOT_VPN` отсутствие | Системный флаг |
| Имя интерфейса `tun0` в ядре | Ядро присваивает, изменить — только root + `ip link` |
| Факт `VpnService.Builder.establish()` | Системный вызов |

**С root/Magisk:** возможно через LSPosed + модуль VPNHide; патч системных API через Xposed.

**На GrapheneOS:** разработчики признают — скрыть TRANSPORT_VPN невозможно без изменения AOSP.

### 5.2 Отключение xray gRPC API ⭐ ОБЯЗАТЕЛЬНАЯ МЕРА

**Конфиг xray БЕЗ API** — просто не включать секцию `"api"`:

```json
{
  "log": {"loglevel": "warning"},
  "inbounds": [{
    "tag": "socks-in",
    "protocol": "socks",
    "listen": "127.0.0.1",
    "port": 10808,
    "settings": {"auth": "noauth", "udp": true}
  }],
  "outbounds": [{
    "tag": "proxy",
    "protocol": "vless",
    "settings": { "vnext": [{
      "address": "your-server.com",
      "port": 443,
      "users": [{"id": "uuid", "flow": "xtls-rprx-vision"}]
    }]},
    "streamSettings": {
      "network": "tcp",
      "security": "reality",
      "realitySettings": { ... }
    }
  }],
  "routing": {"rules": [...]}
}
```

**Эффект отключения API:**
- Порт 10085 не открывается — обнаружение по localhost сканированию невозможно
- `xray api` CLI-команды перестают работать
- Динамическое добавление пользователей через API недоступно
- Статистика через gRPC недоступна

**sing-box:** удалить блок `"v2ray_api"` из `experimental`.

**Clash/Mihomo:** удалить `external-controller` или установить `secret`.

### 5.3 Нестандартные порты — рандомизация

**Насколько помогает:** снижает риск случайного обнаружения, НЕ защищает от полного сканирования `127.0.0.1:1-65535`.

```bash
# Рандомизация при старте (wrapper-скрипт):
RANDOM_PORT=$((RANDOM % 55000 + 10000))
# Подстановка в конфиг перед запуском xray
```

**Дополнительные меры:**
- Bind только к `127.0.0.1` (не `0.0.0.0`)
- Аутентификация на SOCKS/HTTP inbound (если включён)
- TUN-режим без proxy inbound — полностью убирает порты

### 5.4 TUN-режим — от чего защищает, от чего нет

**Защищает:**
- Скрывает SOCKS5/HTTP порты (нет локальных proxy inbound)
- Весь трафик идёт через TUN — нет утечек через системный прокси
- Защищает от `http.proxyHost` детекции

**НЕ защищает:**
- `TRANSPORT_VPN` flag остаётся
- Имя интерфейса `tun0` видно
- Если xray API включён — порт 10085 остаётся открытым
- Пакет VPN-приложения по-прежнему установлен

### 5.5 Android 10+ ограничения /proc/net/ — работает

| Версия Android | Статус |
|---|---|
| До Android 9 | Любое приложение читает `/proc/net/tcp` — видит все соединения |
| Android 10+ (targetSdk ≥ 29) | SELinux `proc_net`: только VPN-домен читает `/proc/net/` |
| Native code (NDK) | SELinux работает на уровне ядра — блокирует независимо от способа вызова |

**Исключения:** системные приложения с platform-key подписью.

**Вывод:** изоляция реально работает для приложений из Play Store с targetSdk ≥ 29.

### 5.6 Firewall на уровне устройства без root — ограничено

**Принципиальная проблема:** на Android одновременно работает только одно VPN-приложение. Сам VPN-клиент занимает VpnService-слот → нельзя запустить NetGuard поверх.

**Альтернативы:**
- **Встроенный firewall в VPN-приложение:** NekoBox, Hiddify фильтруют трафик к 127.0.0.1 внутри TUN
- **ShizuWall (Android 11+, Shizuku):** no-root firewall через ADB-делегирование
- **Android Work Profile:** MDM-политики для ограничения сетевого доступа

**Вывод:** полная блокировка localhost-соединений для сторонних приложений без root **невозможна** стандартными средствами. Лучшая стратегия — убрать саму точку входа (отключить API-порт).

### 5.7 Рандомизация MTU против MTU-fingerprinting

**AmneziaWG конфиг:**
```ini
[Interface]
MTU = 1280          # нестандартный MTU
Jc = 4              # junk-пакеты перед handshake (4-30)
Jmin = 40
Jmax = 70
H1 = 1998811297     # случайный magic header (замена фиксированного)
H2 = 1719675119
H3 = 1717128540
H4 = 1177921885
```

**xray fragmentation:**
```json
"outbounds": [{"protocol": "freedom", "settings": {
  "fragment": {
    "packets": "tlshello",
    "length": "100-200",
    "interval": "10-20"
  }
}}]
```

### 5.8 QUERY_ALL_PACKAGES — обход сигнатурной детекции по пакету

**Кастомная сборка с другим package name:**
```
app.hiddify.com → com.example.mediaplayer
```
Эффективно против сигнатурной детекции по имени пакета.

Ограничение: если у детектора есть `QUERY_ALL_PACKAGES` — он всё равно видит package name. Переименование пакета — единственная реальная защита.

### 5.9 Per-app split tunnel — не скрывает факт VPN

Даже если приложение добавлено в bypass:
```java
// Детекция split-tunnel:
for (Network net : cm.getAllNetworks()) {
    if (nc.hasTransport(TRANSPORT_VPN)) {
        // VPN-сеть существует в системе
        // но текущее приложение исключено из неё
    }
}
```
`TRANSPORT_VPN` на системном уровне видна через `getAllNetworks()`.

### 5.10 Сводная таблица эффективности контрмер

| Контрмера | Сложность | Эффективность | Root нужен? |
|---|---|---|---|
| Отключение xray API | Низкая | Высокая | Нет |
| REALITY transport | Средняя | Очень высокая (против DPI) | Нет |
| AmneziaWG (junk + random headers) | Средняя | Высокая (против DPI) | Нет |
| TUN-режим без proxy inbound | Низкая | Средняя (убирает порты) | Нет |
| Рандомный порт при старте | Низкая | Средняя | Нет |
| Высокий нестандартный порт (47000+) | Низкая | Средняя (против ТСПУ) | Нет |
| Смена package name | Средняя | Средняя (против сигнатур) | Нет |

| /proc/net изоляция | Работает само (Android 10+) | Высокая | Нет |

## Блок 6. Что реально работает в 2026

### 6.1 Против ТСПУ (рейтинг устойчивости)

1. **AmneziaWG 2.0** — наиболее устойчив, но старые версии AmneziaWG блокируются
2. **VLESS + Reality на высоких портах (47000+)** — ТСПУ не инспектирует все порты полностью
3. **XHTTP** — следующее поколение транспорта, адресует уязвимости JA3/JA4 и поведенческого анализа
4. **Shadowsocks-2022 с модифицированными настройками** (plain SS заблокирован)

**Антидетекционные практики против ТСПУ:**
- Отказ от порта 443 → случайные высокие порты
- Пустой SNI или случайный SNI
- Отказ от имитации Chrome fingerprint в пользу нейтрального Go-fingerprint
- Ротация серверов и портов
- Хосты с низким трафиком (большой трафик быстрее детектируется)
- nfqws с fake-пакетами (bolvan) — для обхода по первым пакетам сессии

### 6.2 Против App-level детекции

**Что нельзя скрыть (без root):**
- `TRANSPORT_VPN` flag
- Имя интерфейса `tun0`
- Факт установленного VPN-приложения (если у детектора `QUERY_ALL_PACKAGES`)

**Что можно сделать:**
1. **Отключить xray/Clash API** — убрать порты с localhost
2. **TUN-режим** — убрать SOCKS/HTTP proxy порты
3. **Рандомизировать порт** при каждом старте
4. **Кастомный package name** — против сигнатурной детекции
5. **VLESS + Reality** — против сетевого DPI

### 6.3 Приложения с лучшей защитой

| Приложение | Ключевые защиты |
|---|---|
| **Hiddify** | REALITY, fragmentation, strict_route TUN, множественные outbounds |
| **NekoBox** | Прямой доступ к sing-box JSON, настраиваемый MTU, FakeIP DNS, AmneziaWG |

## Приложение A. Данные ntc.party — расширенный технический разбор

> Источник: NTC — Networking & Telecom Community (ntc.party)
> Ключевые авторы: ValdikSS, 0ka, bolvan, runetfreedom

### A.1 Архитектура ТСПУ — аппаратная часть

Комплекс АСБИ (автоматизированная система) состоит из четырёх компонентов в разрыве каналов операторов. Поставщики: **RDP.RU** (EcoFilter/EcoHighway), **АО «ДЦА»**. Заказчик: **ГРЧЦ**.

**Компоненты стойки:**

1. **Байпас-панели (L2)** — мониторят балансировщик heartbeat-пакетами. При отсутствии heartbeat → физический bypass, трафик идёт насквозь без фильтрации. Балансировщик жёстко разбит на 2 конвейера (Pipeline 1/2), инженеры переключают перемычками.

2. **Балансировщики EcoHighway — 3.2 Тбит/с, 32 порта.** Распределяют трафик по фильтрам хэшированием по кортежу **(SrcIP + DstIP + Protocol)**. Это гарантирует: все пакеты одной TCP/UDP-сессии в обе стороны попадают на один и тот же фильтр и одно ядро процессора — критически важно для сборки сессий.

3. **DPI-Фильтры EcoFilter (Intel Xeon)** — работают на L2 (**без маршрутизируемых IP-интерфейсов**). DPI L4-L7, сигнатурный анализ, отслеживание TCP-сессий.

4. **Сегмент управления** — VPN-шлюзы для связи с ЦСУ в Москве. Провайдер правила менять не может — только смотреть. Таймаут команды управления: **15 минут** (возможный триггер для части блокировок).

**Два типа установки:**
- **Тип А (In-line)** — в разрыв сети, основной
- **Тип Б (Эшелон)** — на стыках с крупными операторами; поддерживает **Port Redirect** (принудительное заворачивание трафика на фильтры) и блокировку через BGP

> Источник: ntc.party/t/22307 (17.02.2026) + github.com/DanielLavrushin/tspu-docs (24 главы)

### A.2 Механика блокировки VPN-протоколов — точные детали

**Общий механизм (ValdikSS, ntc.party/t/5124, 07.08.2023):**
> *"Для всех VPN-протоколов, кроме L2TP, сначала выполняется обнаружение протокола, затем 'проверка' (мониторинг нескольких пакетов в сессии), затем разрыв TCP или блокировка UDP по кортежу srcip-srcport-dstip-dstport."*

| Протокол | Точный триггер |
|---|---|
| L2TP | Блок первых пакетов L2TP Control Message (до порта 1701) |
| IPsec | Блокировка UDP после нескольких пакетов установления сессии |
| PPTP | TCP RST после Start-Control-Connection-Reply, до GRE |
| OpenVPN UDP/TCP | Блок после нескольких P_DATA пакетов после хэндшейка |
| WireGuard | Блокировка UDP после **5 Transport Data пакетов** с сервера |

**Важно:** блокируются только **исходящие соединения** из РФ на зарубежные серверы. VPN из-за рубежа внутрь РФ — работает.

**DPI смотрит только первые ~6 пакетов** (bolvan, ntc.party/t/5239):
> *"DPI смотрит в пределах 6 первых пакетов по 'соединению', потом отпускает и добавляет в allow list."*

Обход: `nfqws --dpi-desync=fake --dpi-desync-any-protocol --dpi-desync-cutoff=d2 --dpi-desync-repeats=10 --dpi-desync-ttl=3`

### A.3 Индивидуальные триггерные блокировки (Active Probing)

С января 2024 ТСПУ применяет **5-минутные блокировки по конкретному IP пользователя** (ValdikSS, ntc.party/t/7318):

При обращении к известным доменам VPN-сервиса (Windscribe: `(api|checkup|assets).windscribe.com`) + последующем подключении к неизвестному IP по VPN-порту → этот IP блокируется на 5 минут как потенциальный VPN-сервер.

По IPv6 — SNI-блокировка постоянная (не временная). Regexp для Windscribe:
```
(api|checkip|assets)\.[0-9a-f]{40}\.com
```

> *"Метод используется для выявления новых VPN-серверов: обратился на домены Windscribe → в течение 5 минут подключился к неизвестному IP по WireGuard → этот IP скорее всего является VPN-сервером."*

### A.4 Shadowsocks — механика детекции

**Счётчик соединений к IP:port** (ntc.party/t/6142, usnevst):
> *"У SS нет мультиплексирования. Считается число подключений к паре адрес:порт. При открытии браузером YouTube число подключений к SS достигает 18, тогда как браузер открывает максимум 12."*

Блок при превышении лимита характерного для обычного браузера (>12–18 соединений к одному IP:port).

### A.5 Reality — модификация ClientHello ТСПУ

**Ntc.party/t/22934 (18.03.2026):** ТСПУ активно модифицирует ClientHello пакет:
- IP TOS: с `0x00` меняется на `0x28`
- IP Length, TLS Length, Client Random изменяются
- **Cipher Suites переставляются местами**, Extensions не совпадают

Результат: xray сервер видит невалидный ClientHello и отбрасывает соединение:
```
[Info] transport/internet/tcp: REALITY: processed invalid connection from ...: failed to read client hello
```

Это **активное Man-in-the-Middle вмешательство**, а не пассивная блокировка.

### A.6 «Сибирская блокировка» и gRPC-транспорт

Апрель 2026, массовые проблемы с VLESS+Reality на провайдерах МО и Москвы.

**«Сибирская блокировка»** — специфический паттерн: один ТСПУ блокирует, другой параллельный — нет.

Решение (Dreaght + 0ka, ntc.party/t/23653):
> *"Лечится транспортным мультиплексированием gRPC. Разделение входного/выходного IPv4 И транспорт gRPC."*

**gRPC-транспорт напрямую (не через CDN) в апреле 2026 работает надёжнее** чем TCP-based Reality на порту 443.

### A.7 Блокировка SSH по сигнатуре первых пакетов

**Ntc.party/t/13275 (ноябрь 2024):**

ТСПУ блокирует SSH-авторизацию **по ключу** (RSA, ED25519), но пропускает по паролю. На IP-адресах, ранее ассоциированных с VPN-протоколами.

(shatteredquartz, пост 9):
> *"Если первые два пакета от клиента были больше определённого размера — соединение разрывалось, но только на портах, отличных от 22, 80 и 443. Клиент отправлял первые два пакета размером больше 24 и 8 байт — соединение после них отрезалось."*

Порты 22, 80, 443 имеют особый режим обработки — не блокируются по этой сигнатуре.

### A.8 ECH — дополнительный вектор блокировки

**Ntc.party/t/19771 (28.09.2025):**

Firefox с DoH автоматически включает **ECH (Encrypted Client Hello)**. ТСПУ блокирует ECH-хэндшейки к сайтам из блэклиста.

Workaround: `network.dns.echconfig.enabled = false` в Firefox.

curl не поддерживает ECH → проходит через ТСПУ где Firefox не проходит.

### A.9 VK MAX — детали из ntc.party (тред 22584, 266 постов)

Бинарный протокол поверх TCP, команды длиной 10 байт:
```
0a|0100|01|0006|01|000087|data
```
- Поле `operator`: PLMN текущей сети (`'operator': '25001:MTS'`) — **берётся из текущей сети, а не SIM-карты**
- `callDontUseVpnForRtp` — конфиг с сервера для C-кода: **обход системного VPN при звонках**
- Все запросы идут на одно соединение с `api.oneme.ru`

Механизм необнаруживаемости: данные смешаны с основным трафиком мессенджера — невозможно заблокировать аналитику без блокировки приложения.

### A.10 МТС — детекция через поведенческий анализ трафика

**Ntc.party/t/22880 (16.03.2026):**

МТС вводит плату 87 руб./день за «файлообменные сети» (= VPN/P2P трафик):

> *"Операторы видят, что значительная часть трафика одного пользователя идёт к одному IP на протяжении значительного времени. VPN полностью ломает схему GGC (Google Global Cache) — кэши внутри сети оператора."*

Метод: **поведенческий — длительный трафик к одному IP:port** = характерный паттерн VPN-туннеля.
